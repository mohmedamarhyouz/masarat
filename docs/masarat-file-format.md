# Masarat File Formats

Masarat reads UTF-8 JSON files with either the `.masarat` or `.json` extension. The application contains no AI service: a file can be written manually or generated by any external AI assistant.

## Required root fields

```json
{
  "format": "masarat",
  "schemaVersion": "1.0",
  "project": {},
  "constraints": [],
  "nodes": [],
  "connections": [],
  "tasks": [],
  "assumptions": [],
  "checkpoints": [],
  "realityEvents": [],
  "changes": [],
  "versions": []
}
```

The complete working example is available at `public/examples/g3-pro-plan.masarat`.

## Stable identifiers

Every node, task, connection, assumption, checkpoint, event, change, and version needs a unique string `id`. An external AI must retain an existing identifier when the element still represents the same thing. This is how Masarat keeps completed task states, actual costs, and notes while importing an updated plan.

## Qualitative estimates

`probability`, `confidence`, and impact values accept:

- `low`
- `medium`
- `high`
- `unknown`

Do not invent precise numerical probabilities without real data.

## Node types

`situation`, `goal`, `decision`, `option`, `task`, `scenario`, `risk`, `checkpoint`, `change`, `outcome`, and `lesson`.

Each node requires `id`, `type`, `title`, and a numeric horizontal `stage` starting at zero.

## Updating a project

1. Record a major change in the application.
2. Export the AI update package from the Versions page.
3. Ask an external assistant to return only a valid Masarat 1.0 project.
4. Import the returned file.
5. Masarat shows the structural difference and preserves local task progress by matching stable IDs.

## Life Pack 2.0

A Life Pack imports or updates several life areas, goals, projects, and generic metrics in one file:

```json
{
  "format": "masarat",
  "schemaVersion": "2.0",
  "packageType": "life-pack",
  "title": "My Life Plan",
  "exportedAt": "2026-07-14T20:00:00.000Z",
  "areas": [],
  "goals": [],
  "projects": [],
  "metrics": [],
  "metricEntries": [],
  "reviews": []
}
```

Projects inside a Life Pack remain valid Masarat 1.0 projects. Before merging, the application previews new and updated entities. Existing projects are merged by stable project and task IDs so completed states, actual costs, notes, reality events, and version history remain local.

Life-area objects require `id`, `name`, `icon`, a six-digit hex `color`, `status`, `order`, `createdAt`, and `updatedAt`. Goals require `id`, `areaId`, `title`, `status`, progress from 0–100, `createdAt`, and `updatedAt`.

## Full backup 1.0

Settings can export a complete local backup:

```json
{
  "format": "masarat-backup",
  "schemaVersion": "1.0",
  "exportedAt": "2026-07-14T20:00:00.000Z",
  "data": {
    "projects": [],
    "lifeAreas": [],
    "goals": [],
    "metrics": [],
    "metricEntries": [],
    "reviews": []
  }
}
```

Restoring a backup replaces the current local database only after validation and explicit confirmation. Export a fresh backup before restoring if the current state may be needed later.

## Privacy note

IndexedDB is local storage, not encryption. Do not place passwords, banking credentials, or identity-document scans in Masarat.
